# PH15 - Dynamic Role Selection Login & Module Management System

## Overview
This phase implements three user-requested features:
1. **Dynamic role selection at login** — roles are hidden until credentials are validated; only roles assigned to the authenticated user are shown.
2. **Multi-role user support** — a single user can be assigned multiple roles and explicitly chooses which role to use each session.
3. **Module management system** — SuperAdmin can enable/disable modules globally; both SuperAdmin and Admin can configure per-user module access.

---

## Feature 1 & 2: Dynamic Role Selection Login (Two-Step UX)

### Problem
The original login form showed a static role dropdown listing all system roles before the user entered credentials. This leaked role names and didn't support per-user role assignment or selection.

### Solution: AJAX Two-Step Login Flow

**Step 1 — Credentials only:**
- User enters email and password and clicks **Continue**.
- An AJAX POST is sent to `/Account/ValidateCredentials`.
- The server validates the password (without signing in) via `UserManager.CheckPasswordAsync`.
- Returns `{ success: true, roles: ["Admin", "Doctor"] }` or `{ success: false }`.
- If successful, the credential fields are hidden and the role selector appears.

**Step 2 — Role selection:**
- The `<select>` is dynamically populated with only the user's assigned roles.
- If only one role is available it is pre-selected.
- User clicks **Sign In** which submits the full form (email, password, selectedRole).

### Key Code: ValidateCredentials Endpoint
```csharp
[HttpPost, AllowAnonymous, ValidateAntiForgeryToken]
public async Task<IActionResult> ValidateCredentials([FromForm] string email, [FromForm] string password)
{
    var user = await _userManager.FindByEmailAsync(email);
    if (user == null || !await _userManager.CheckPasswordAsync(user, password))
        return Json(new { success = false });
    var roles = await _userManager.GetRolesAsync(user);
    return Json(new { success = true, roles });
}
```

### Key Code: Active Role Session Storage
```csharp
HttpContext.Session.SetString("ActiveRole", selectedRole);
```

### Role Priority Fallback (PickPrimaryRole)
If the user bypasses role selection, `PickPrimaryRole` applies this priority order:
`SuperAdmin → Admin → Doctor → Nurse → Pharmacist → Accountant → Receptionist → Patient`

## Feature 3: Module Management System

### Problem
No mechanism existed to turn off system modules globally or restrict modules per user.

### Data Model

**`SystemModule`** (in `Models/RBAC.cs`):
```
Id, Key (unique), DisplayName, Icon, Description, IsGloballyEnabled (default true), SortOrder
```

**`UserModuleAccess`** (in `Models/RBAC.cs`):
```
Id, UserId (FK), ModuleId (FK), IsEnabled, UpdatedAt, UpdatedByUserId
Unique index on (UserId, ModuleId)
```

A user cannot have a globally-disabled module individually enabled. `ModuleService.SetUserModuleAccessAsync` returns an error if this is attempted.

## Runtime Compatibility Note
- The login page relies on jQuery for the two-step credential validation flow.
- If Content-Security-Policy is enabled, jQuery must be loaded from an allowed host (current layouts use `https://cdn.jsdelivr.net/npm/jquery@3.7.1/dist/jquery.min.js`, which matches the configured CSP `script-src`).
- The shared login view is used from multiple routes, so the form action is hard-pinned to `/Account/Login` to avoid route ambiguity with `PatientPortal/Account/Login`.
- Because the form action is explicit, antiforgery is added explicitly with `@Html.AntiForgeryToken()`.
