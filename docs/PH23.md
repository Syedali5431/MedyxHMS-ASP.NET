# PH23 - Password Governance and Accounts Approval Workflow

## Objective
Implement governed account lifecycle controls:
1. role-constrained admin password reset authority
2. approval-gated activation for newly signed-up users

## Scope Delivered
- Added self-service signup route (`Account/Register`) that creates:
  - user account with `IsActive = false`
  - pending `AccountApprovalRequest`
- Added Admin/SuperAdmin section `Accounts Approval` with pending request queue and approve/reject actions.
- Approval by either Admin or SuperAdmin now activates the target account immediately.
- Rejection now requires a reason and stores it in approval request notes.
- Added `Password Management` section for Admin/SuperAdmin to reset user passwords with role restrictions.

## Password Authority Rules
- Admin can reset passwords for non-SuperAdmin accounts.
- SuperAdmin can reset passwords for all accounts, including SuperAdmin.
- SuperAdmin-target resets are blocked for non-SuperAdmin actors.
- Existing staff password reset path now enforces the same role rule.

## Data Model and Compatibility
- Added `AccountApprovalRequest` entity and `DbSet`.
- Added DB bootstrap/upgrade SQL to create `AccountApprovalRequests` table with indexes for existing deployments.

## UX Changes
- Login page now includes `Sign up for approval` link.
- Inactive login feedback now distinguishes pending approval vs rejected account request.
- Accounts Approval reject action shows inline validation feedback when reason is missing.
- Main authenticated navbar for Admin/SuperAdmin now includes:
  - Accounts Approval
  - Password Management

## Validation
- Build and test suite executed successfully after implementation.
- No diagnostics errors in modified files.
