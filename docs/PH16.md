# PH16 - Role Enforcement Hardening and UAT Automation

## Overview
This phase closes the remaining operational readiness gaps after licensing, chatbot, export, and user-guide work. The focus is on making role rules explicit, reducing accidental authorization leakage in views, and adding repeatable smoke automation for local validation.

## Delivered Changes

### 1. RBAC Seed Alignment
- Added missing RBAC role seeds for `Accountant` and `Patient`.
- Expanded permission seeding so all non-patient operational roles can add/edit patient details as requested.
- Preserved `Patient` as a restricted role with no staff-management or clinical admin permissions.

### 2. View-Layer Permission Hardening
- Replaced the old `ClaimsPrincipalExtensions.HasPermission` behavior that returned `true` for any authenticated user.
- Added an explicit role/action matrix for:
  - `ManageUsers`
  - `Appointment`
  - `Patient`
- Result: UI action buttons now align much more closely with actual business permissions instead of exposing controls to any signed-in user.

### 3. Report Governance Hardening
- Confirmed report editing operations remain restricted to `Admin` and `SuperAdmin`.
- Applied explicit role guards to report scheduling and delete operations.

### 4. Automated UAT Smoke Pack
Added PowerShell automation under `scripts/`:
- `Invoke-UatSmoke.ps1`
  - builds the web app
  - builds the MedyxHMS-Lic desktop tool
  - runs test project
  - optionally generates a signed `.lic` artifact
  - optionally checks HTTP endpoints if a base URL is available
- `Invoke-LicenseToolAutomation.ps1`
  - automates interactive MedyxHMS-Lic console input for key generation and license creation
- `UAT-Smoke.config.template.json`
  - template config for smoke execution inputs

### 5. Regression Coverage
- Added integration coverage for `ClaimsPrincipalExtensions.HasPermission` to prevent regressions in role-gated UI actions.
