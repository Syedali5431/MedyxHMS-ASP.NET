# MedyxHMS Licensing System — Technical Reference

**Document:** MedyxHMS-LIC.md  
**Audience:** System Administrators, SuperAdmins, Medyx Vendor Team  
**Last Updated:** 2026-06-24  

---

## Overview

The Medyx HMS licensing system uses **asymmetric cryptography** (RSA-4096) to generate tamper-proof, signed license files that control what modules a hospital can access and for how long.

The system has **two components:**

| Component | Location | Purpose |
|-----------|----------|---------|
| `MedyxHMS-Lic` | Vendor-side console tool | Generate RSA key pairs; generate signed `.lic` files |
| `MedyxHMS` (ASP.NET) | Deployed at the hospital | Validate and enforce the license at runtime |

**Core security principle:** The **RSA private key never leaves vendor-controlled systems.** Only the public key modulus and exponent are embedded in the deployed application (via upload). License files can only be generated by someone holding the private key; they can only be verified by someone holding the matching public key.

## Part 1 — Key Generation

### Purpose
Generates an RSA-4096 key pair. The private key is used by the vendor to sign future license files. The public key is given to the hospital (uploaded into the ASP.NET application) to enable license verification.

### How to Run
```
MedyxHMS-Lic.exe
> Select option 1: Generate Key Pair
```

### What Happens Internally

1. `RSA.Create(4096)` — generates a 4096-bit RSA key pair.
2. A `KeyId` is generated: 256-bit random hex (32 bytes, hex-encoded) — this is a collision-resistant unique identifier for the key pair.
3. **Public Key JSON** is built with the following fields:

```json
{
  "KeyId": "<256-bit random hex>",
  "Algorithm": "RSA-4096",
  "KeySize": 4096,
  "CreatedAtUtc": "2026-04-22T10:00:00Z",
  "ModulusHex": "<RSA modulus in hex>",
  "ExponentHex": "<RSA public exponent in hex>",
  "VerificationKey": "<SHA256 fingerprint>"
}
```

4. **VerificationKey** computation:
   ```
   VerificationKey = SHA256("MEDYX-VERIFY|{ModulusHex}|{ExponentHex}")
   ```
   This is a cryptographic fingerprint of the public key. It is embedded in both the key file and the license payload. When the hospital uploads the public key, the system recomputes this fingerprint and compares it — ensuring the public key presented matches the one used to generate the license.

5. **Private Key JSON** extends the public key JSON with the RSA private parameters:

```json
{
  ... (all public key fields),
  "DHex": "<private exponent>",
  "PHex": "<prime P>",
  "QHex": "<prime Q>",
  "DPHex": "<DP = D mod (P-1)>",
  "DQHex": "<DQ = D mod (Q-1)>",
  "InverseQHex": "<InverseQ = Q^-1 mod P>"
}
```

### Output Files

| File | Contents | Who Gets It |
|------|----------|-------------|
| `medyxhms-public-key-{keyId}.json` | Public modulus, exponent, verification key | Shared with hospital (used during upload) |
| `medyxhms-private-key-{keyId}.json` | Full RSA private parameters | Stays on vendor machine ONLY |

### Private Key File Hardening (Windows)
After writing the private key file, the tool applies Windows ACL security:
- Removes all inherited permissions (inheritance disabled).
- Removes all existing access control entries.
- Grants **FullControl** only to the **current user** (the operator running the tool).
- Applied via `FileSecurity` / `SetAccessControl` (.NET).

> **CRITICAL:** The private key file must never be emailed, uploaded, or otherwise transmitted. It must stay on the vendor's controlled system. Compromise of the private key would allow unauthorised license generation.

## Part 2 — License File Generation

### Purpose
Creates a signed `.lic` file that authorises a specific hospital (tenant) to use Medyx HMS with defined module entitlements for a defined time period.

### How to Run
```
MedyxHMS-Lic.exe
> Select option 2: Create License
> Enter path to private key JSON file: medyxhms-private-key-{keyId}.json
```

### Interactive Prompts

| Prompt | Input | Validation |
|--------|-------|------------|
| TenantId | Unique identifier for the hospital (e.g., hospital GUID or name) | Non-empty |
| Expiry | Choose: 1-month trial / 1 year / 2 years / 3 years / Custom date | Min 1 month from today |
| MaxConcurrentUsers | Maximum simultaneous active users allowed | Positive integer |
| Module selection | Y/n for each optional module | See module catalogue below |

### License Payload Structure

The tool builds a `LicensePayload` object with the following fields (in canonical key order):

```json
{
  "ProductName": "MedyxHMS",
  "TenantId": "<hospital identifier>",
  "LicenseId": "<new GUID — unique per license>",
  "IssuedAt": "<UTC timestamp of generation>",
  "ExpiresAt": "<UTC timestamp of expiry>",
  "MaxConcurrentUsers": 50,
  "VerificationKey": "<copied from private key file — SHA256 fingerprint>",
  "LicensedModules": ["Ambulance", "Appointment", "Billing", ...],
  "Nonce": "<32-byte cryptographically random hex>"
}
```

**Canonical rules:**
- `LicensedModules` is sorted alphabetically, trimmed, and deduplicated.
- JSON key order is always deterministic (fixed field order as above) to ensure the signed canonical string is identical regardless of serialiser.
- The `Nonce` (32-byte random hex) is generated fresh each time to prevent replay attacks.

### Signing Process

1. Canonical JSON string is produced from `LicensePayload` (UTF-8 encoded).
2. RSA-SHA256 PKCS#1 v1.5 signature is computed:
   ```
   SignatureHex = RSA.SignData(UTF8(canonicalJson), SHA256, PKCS1)
   ```
3. A `SignedLicenseFile` object is assembled:

```json
{
  "Payload": { ... LicensePayload ... },
  "Algorithm": "RSA-SHA256",
  "SignatureHex": "<hex-encoded RSA signature>"
}
```

### Output File Format

1. `SignedLicenseFile` is serialised as **minified JSON**.
2. Minified JSON is **Base64-encoded**.
3. The file is written as:
   ```
   MEDYX-LIC-V1:{base64encodedstring}
   ```
4. Saved to: `MedyxHMS.lic`

### Console Output

After generation, the tool displays:
- Output file path (absolute path to `MedyxHMS.lic`)
- `LicenseId` (GUID) — **record this** for your audit trail

## Part 3 — Uploading the License (SuperAdmin)

The hospital SuperAdmin installs the license in two steps.

### Step 1: Apply Public Key Configuration
1. Log into Medyx HMS as SuperAdmin.
2. Navigate to **License Management** from the navigation.
3. In the **Key Configuration** section, enter:
   - **Modulus Hex** — from the public key JSON file (`ModulusHex` field)
   - **Exponent Hex** — from the public key JSON file (`ExponentHex` field)
   - **Verification Key** — from the public key JSON file (`VerificationKey` field)
4. Click **Apply Key Configuration**.
5. The system stores these values securely.

> Public key values should be provided to the SuperAdmin by the vendor. They come from the `medyxhms-public-key-{keyId}.json` file.

### Step 2: Upload the License File
1. Still on the **License Management** screen.
2. Click **Upload License**.
3. Select the `MedyxHMS.lic` file provided by the vendor.
4. Click **Upload / Apply**.

### What Happens on Upload

1. The file is read; the `MEDYX-LIC-V1:` prefix is stripped.
2. The base64 string is decoded to obtain the minified JSON.
3. The `SignedLicenseFile` is deserialized.
4. The canonical JSON string is regenerated from the `Payload`.
5. The stored public key (Modulus + Exponent) is loaded and used to verify:
   ```
   RSA.VerifyData(UTF8(canonicalJson), SignatureHex, SHA256, PKCS1)
   ```
6. The `VerificationKey` in the payload is compared to the stored `VerificationKey` — they must match.
7. If verification passes:
   - The license record is updated with the new `ExpiresAt`, `LicensedModules`, `MaxConcurrentUsers`, and `TenantId`.
   - Changes take effect immediately.
8. If verification fails: the upload is rejected with an error message (see Troubleshooting).

### Post-Upload Behaviour
- The public key fields are cleared (one-time key consumption) to prevent reuse of the same key pair for a future upload. A new key pair must be generated for the next license cycle.
- All licensing actions are recorded in the Audit Trail.

## Part 4 — Runtime Enforcement

### Module Gating
- When any user attempts to access a module URL, the system checks whether the module is present in the `LicensedModules` list.
- If the module is **not licensed**, the request is blocked.
- Non-SuperAdmin users see an "Access Denied" or "Module Not Available" screen.
- **SuperAdmin is exempt** from module gating and can always access all system areas.

### Expiry Enforcement
A Quartz.NET background job runs a **daily license check**:
- If the license is still valid: all modules function normally.
- If the license has expired:
  - All staff and admin module access is blocked.
  - Blocked users are redirected to a license-expired information screen.
  - **Exempt users:** SuperAdmin (can always log in to upload a new license) and Patients (Patient Portal remains accessible).

### Pre-Expiry Email Reminders
- **5 days before expiry:** The system automatically sends a license renewal reminder email to all registered users.
- Duplicate suppression: the reminder is only sent once per expiry cycle (once per license period) to avoid spam.
- The SuperAdmin can also manually trigger a reminder resend from the License Management screen.

### License Renewal
To renew (extend) the license:
1. The vendor generates a new `MedyxHMS.lic` with a new `ExpiresAt` date.
2. A new key pair must be generated (one-time key consumption policy).
3. The SuperAdmin applies the new public key configuration and uploads the new `.lic` file.
4. The new expiry takes effect immediately.

> Available extension periods: +1 year, +2 years, +3 years only. These are enforced at generation time by the vendor tool. Trial extensions (1-month) require a full new license rather than a renewal.

## Part 5 — Security Model

| Security Property | How It Is Achieved |
|-------------------|--------------------|
| Tamper resistance | RSA-SHA256 signature over canonical payload — any byte change invalidates the signature |
| Deterministic signing | Canonical JSON with fixed key order ensures the exact same bytes are signed and verified |
| Replay prevention | `Nonce` (32-byte random) is unique per license file — the same `.lic` cannot be used for a different tenant |
| Key binding | `VerificationKey` (SHA256 fingerprint of public key) is embedded in the payload and re-checked at upload |
| One-time key use | Public key fields are cleared after a successful upload; the same key pair cannot be reused |
| Private key protection | Windows ACL on the private key file (owner-only FullControl); key never leaves vendor systems |
| Audit trail | All license actions (upload, renewal, expiry, manual reminder) are logged in the system Audit Trail |

## Part 6 — Troubleshooting

### Upload rejected: "Verification key mismatch"
**Cause:** The `VerificationKey` in the `.lic` file does not match the `VerificationKey` entered in the Key Configuration step.  
**Fix:** Ensure you are using the public key JSON from the same key pair used to generate the `.lic` file. Copy all three values (Modulus, Exponent, VerificationKey) exactly from the correct `medyxhms-public-key-*.json` file.

### Upload rejected: "Signature verification failed"
**Cause:** The `.lic` file content has been altered (manually edited or corrupted).  
**Fix:** Request a freshly generated `MedyxHMS.lic` from the vendor. Do not attempt to edit `.lic` files manually.

### Upload rejected: "Reused key"
**Cause:** The same public key values were used for a previous upload (one-time key consumption policy).  
**Fix:** Generate a new RSA-4096 key pair using `MedyxHMS-Lic` Option 1. Use the new key pair to generate the new `.lic` file.

### Module still locked after successful upload
**Cause:** The module was not selected (Y) during `.lic` file generation.  
**Fix:** Generate a new `.lic` file, ensuring you select Y for the required module(s) during the interactive module selection step.

### License expired — staff cannot log in
**Cause:** The current date has passed the `ExpiresAt` date in the license.  
**Fix:** Contact the vendor to obtain a new `MedyxHMS.lic`. The SuperAdmin can still log in (exempt from expiry enforcement) to apply the new license.

### Lost the private key file
**Cause:** Private key file deleted or inaccessible.  
**Fix:** You cannot recover a lost RSA private key. Generate a new key pair using Option 1. Apply the new public key configuration in the ASP.NET application. Generate a new `.lic` using the new key pair.

## Appendix — File and Field Reference

### LicensePayload Fields

| Field | Type | Description |
|-------|------|-------------|
| ProductName | string | Always `"MedyxHMS"` |
| TenantId | string | Unique hospital identifier |
| LicenseId | GUID | Unique per generated `.lic` file |

| ExpiresAt | UTC datetime | When the license expires |
| MaxConcurrentUsers | int | Max simultaneous logged-in users |
| VerificationKey | string | SHA256 fingerprint of the signing public key |
| LicensedModules | string[] | Sorted, deduplicated list of licensed module names |

### .lic File Format

```
MEDYX-LIC-V1:{base64encodedJson}
```

Where the base64-decoded JSON has the structure:
```json
{
  "Payload": { ... LicensePayload ... },
  "Algorithm": "RSA-SHA256",
  "SignatureHex": "..."
}
```
